Decoding Claude Code

What actually happens when you ask an AI to fix your code? Where does the thinking happen? Does your code leave your machine? And why is it eating your RAM?

A technically precise, cricket-flavoured guide to the architecture of Claude Code.

01 — The Split

Your Machine vs. Anthropic's Servers

Here is the single most important thing to understand about Claude Code: it is two completely separate systems pretending to be one.

When you install Claude Code and start chatting with it in your terminal, you might imagine that a tiny AI has moved into your laptop. That is not what has happened. What has actually happened is that a very capable middleman has set up shop on your machine, and it has the AI's phone number.

Your machine runs a TypeScript application built on the Bun runtime. It is roughly 512,000 lines of code.src It handles everything physical: reading your files, running shell commands, searching your codebase with ripgrep, rendering the terminal interface (built with React and a library called Ink), and managing permissions so Claude doesn't accidentally rm -rf your life's work.

Anthropic's servers run the actual AI model — Sonnet, Opus, or Haiku, depending on your configuration. All reasoning, planning, code understanding, and decision-making happens there.docs The model never touches your files directly. It never executes a single command on your machine. It can only ask the local client to do things on its behalf.

The communication between the two happens over HTTPS (TLS-encrypted) via Anthropic's Messages API.docs The model's response streams back as Server-Sent Events, so you see text appearing in real-time rather than waiting for the full response.

Think of it like a Test match. The batsman (the local CLI) is the one at the crease — physically present, playing the shots, running between wickets. The captain sitting in the dressing room (the AI model on Anthropic's servers) is watching the match on a TV feed, sending field placement instructions via walkie-talkie. The captain is the brains of the operation. But the captain never sets foot on the pitch and never touches the ball. The batsman handles all the physical action; the captain handles all the strategy.

02 — The Agentic Loop

How It Actually Works, Ball by Ball

Claude Code does not just answer questions. It acts — reading files, running commands, editing code — in a continuous loop until the job is done. This loop is the heart of the system.

Here is what happens every time you type a message:

Step 1: The local client assembles the full context — your message, the entire conversation history so far, the system prompt, your project's CLAUDE.md instructions, tool definitions, and any auto-memory. This entire bundle is sent to Anthropic's API as a single request.docs

Step 2: The model processes everything and streams back a response. This response might be plain text ("Here's what I found...") or it might contain tool calls — structured requests like "read the file at src/utils.ts" or "run npm test in the terminal."

Step 3: The local client intercepts each tool call, checks it against the permission system (tools are classified as low, medium, or high risk), and if approved, executes it locally on your machine.docs The model never runs anything — it only asks.

Step 4: The tool result (file contents, command output, search results) is appended to the conversation history.

Step 5: The updated conversation is sent right back to the API. The model sees what happened and decides the next move — maybe another tool call, maybe a final answer.

Step 6: This loop repeats — potentially dozens of times in a single response — until the model produces a reply with no further tool calls. Only then does control return to you.

This is an over in a Test match. The bowler runs in and delivers (prompt sent to API). The batsman reads the delivery and plays a shot (model decides a tool call). The ball races to the boundary rope, a fielder chases it down and throws it back (local client executes the tool and returns the result). The bowler gets the ball back and runs in again (next API call with updated context). Six balls per over, but Claude's overs have no fixed limit — the loop continues until the model declares its innings complete.

Each "turn" may involve multiple tool calls — Claude can read, edit, and test in a single response

A key detail: the model can issue multiple tool calls in a single response. If it needs to read three files, it asks for all three at once, and the local client executes them in parallel. This is how Claude can read a file, edit it, run the tests, and respond with the results — all in what feels like a single "turn."

Claude Code can also spawn sub-agents — separate instances with their own isolated context windows. This is how it parallelizes complex tasks: one sub-agent explores the codebase while another researches a library, and neither clutters the main conversation's memory.

03 — The Data Question

What Leaves Your Machine?

This is the question everyone eventually asks, and the answer requires honesty: yes, your code is sent to Anthropic's servers. There is no way around this.

The AI model runs on Anthropic's servers. For it to read your code, understand your project structure, or suggest edits, it must see the code. And "seeing" means the code is transmitted over the network as part of the API request. There is no local model, no on-device inference, no clever trick where the model reasons about your code without actually receiving it.

However, it is not a wholesale upload. Claude Code does not vacuum up your entire codebase and ship it off. It sends only what is needed for the current conversation:

Think of the TV broadcast of a cricket match. The cameras (API requests) show the dressing room captain (the model) what is happening on the pitch. The captain cannot set field placements for a delivery he cannot see. But the cameras are selective — they show the action at the crease, not every vendor in the parking lot. Similarly, Claude sees only the files it explicitly reads, not your entire hard drive.

▲ Sent to Anthropic's servers

  • Your prompts and Claude's responses (the full conversation history)
  • Contents of files that Claude explicitly reads (not all files — only the ones Claude's tools open)
  • Output of shell commands Claude runs (test results, build output, etc.)
  • Your project's CLAUDE.md instructions and auto-memory
  • Tool definitions (so the model knows what tools are available)
  • Telemetry data — usage patterns and latency metrics (no code or file paths)docs

▼ Stays on your machine

  • Your filesystem (files are not uploaded to storage — they're included in API context only when read)
  • All tool execution (bash commands, file edits, git operations run locally)
  • Session files and conversation history (saved locally for resume/fork)
  • Checkpoint snapshots (file state before edits, stored locally for rollback)
  • Permission configurations and settings
  • Files Claude has not read — if Claude never opens a file, its contents never leave your machine
  • Databases, running applications, environment variables (unless you share them in conversation)

Only what Claude reads and the conversation itself are transmitted — everything else stays local

A nuance worth understanding: when Claude reads a file, the file contents become part of the conversation context sent in the API request. They are not uploaded to a separate file store. They exist in the request payload, encrypted in transit via TLS, and are subject to Anthropic's data retention policies (which we'll cover next).docs Once the conversation ends and the retention period passes, those file contents are not persistently stored on Anthropic's side.

04 — Security & Privacy

Stadium Security: Who Gets in Where

If your code is leaving your machine, the obvious follow-up is: how is it protected? The answer depends on which tier of service you're using.

Think of it as stadium access. General Admission (Free/Pro/Max) gets you into the ground with standard security checks. The VIP Enclosure (Team/Enterprise) adds dedicated security, stricter access controls, and your own private viewing area. The Players' Enclosure (API with Zero Data Retention) is the most restricted — once the match is over, even the CCTV footage is erased.

Certifications and Compliance

Anthropic holds SOC 2 Type II attestation, ISO 27001:2022 and ISO/IEC 42001:2023 (AI Management Systems) certifications.docs HIPAA compliance is available via Business Associate Agreement. Compliance documentation is available through the Anthropic Trust Center.

What About Training?

This is the question people most care about, and the answer is more nuanced than a flat yes or no:

  • Consumer plans (Free, Pro, Max) give users the choice to allow data to be used for training. It is an opt-in setting. If you turn it on, data may be retained up to 5 years in de-identified form.docs
  • Commercial plans (Team, Enterprise, API) do not train on your data by default. You would need to explicitly opt in (e.g., via the Developer Partner Program).docs

Locking It Down Further

For maximum privacy, Claude Code supports environment variables that disable non-essential outbound traffic:

DISABLE_TELEMETRY=1
DISABLE_ERROR_REPORTING=1
CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1

Cloud providers like AWS Bedrock and Google Vertex AI offer an additional layer: your API calls stay within your own cloud VPC, and telemetry/error reporting are disabled by default on these platforms.docs

05 — RAM & Resources

What Your Machine Actually Does (And Eats)

If all the thinking happens on Anthropic's servers, why is Claude Code using 800+ megabytes of your RAM? Fair question.

The local client is not a trivial program. It is a full TypeScript application running on the Bun runtime, rendering a React-based terminal interface, maintaining conversation state, buffering tool results, managing file checksums for rollback, and orchestrating the permission system. All of this takes memory.

Measured baseline: approximately 820 MB of resident memory (RSS) per idle session. Official documentation puts the expected range at 100–2,000 MB depending on what Claude is doing. Reading large files, buffering long command outputs, or spawning sub-agents will push the number higher.

The ground staff at a cricket stadium. They maintain the pitch, manage the scoreboard, operate the floodlights, run the PA system, and keep the crowd fed. Without them, no match happens. But they are not the ones scoring centuries. More ground staff (more RAM) means a better-maintained stadium (smoother experience), but they cannot bat or bowl on your behalf. The quality of Claude's thinking is determined by the model on Anthropic's servers — your RAM just determines whether the local experience stutters or flows.

CLI vs. Desktop App vs. VS Code Extension

Claude Code runs in three different environments, all using the same core engine:

The VS Code extension is architecturally interesting: it embeds the CLI binary and runs a local MCP server that the CLI connects to.docs It is not a reimplementation — it is a bridge. The Desktop app is an Electron shell around the same Bun-based core.docs In all three cases, the agentic loop, tool system, and API communication are identical.

Multiple Sessions

Each Claude Code session runs as a fully independent process. Separate process, separate context window, separate memory. There is no shared state between sessions except your CLAUDE.md, auto-memory files on disk, and — importantly — your API rate limits, which are pooled across all sessions under the same subscription.

With stable versions at ~820 MB each, running 4 simultaneous sessions costs roughly 3.2 GB of RAM — manageable on a 16 GB machine. The practical bottleneck for multiple sessions is usually rate limits, not RAM.

Does More RAM Help?

Mostly no. The quality of Claude's output — its reasoning, code understanding, and suggestions — is determined entirely by the model running on Anthropic's servers. Your local RAM has zero influence on that. The model doesn't run faster because your laptop has 64 GB instead of 16 GB.

Where RAM does matter:

  • Preventing swap thrashing — if your OS starts paging Claude Code's process to disk, you'll feel latency on tool execution and UI rendering
  • Running alongside your IDE — VS Code, JetBrains, Docker, browser tabs, and Claude Code all competing for RAM
  • Multiple concurrent sessions — each session is additive (~820 MB each)
  • Large shell command outputs — buffered in memory before being sent to the API

Recommendation: 16 GB is comfortable for single-session use. 32 GB is prudent if you run multiple sessions or use the Desktop/VS Code versions alongside heavy dev tools.

06 — The Context Window

The Scorecard: What the Captain Sees

The context window is the single most important concept for understanding Claude Code's behaviour and limitations. It is the totality of what the model can see on each turn.

Every time the local client sends a request to Anthropic's API, it sends the entire accumulated context: the system prompt, your CLAUDE.md, auto-memory, all tool definitions, every message in the conversation so far, and every tool result. The model does not "remember" anything between turns — it re-reads the full context on every single API call.

The scorecard. It records every ball bowled, every run scored, every wicket taken throughout the innings. The captain in the dressing room studies the full scorecard before each delivery to understand the state of play. But scorecards have a fixed number of pages. When the innings runs long, the scorers start summarising early sessions into compressed notes — "opening partnership of 87 in 22 overs" instead of ball-by-ball detail. That is exactly what context compaction does.

The biggest consumers of context space are tool results (every file Claude reads and every command output) and the conversation history (every message back and forth). The system prompt, CLAUDE.md, and tool definitions are relatively fixed overhead.

Context Compaction

Claude Code currently supports a context window of up to 200,000 tokens per turn. When the accumulated context approaches this limit, the system automatically triggers compaction:

  • First pass: Old tool outputs are cleared (they are the most expendable — the model has already seen and reasoned about them)
  • Second pass: The conversation is summarised into a compressed form, preserving key decisions and recent context while discarding detailed early exchanges
  • Result: The context is brought back within budget, and the conversation continues seamlessly

This is why very long Claude Code sessions can sometimes "forget" instructions you gave early on — those instructions may have been compacted away. It is also why the CLAUDE.md file is valuable: it is re-injected on every turn and survives compaction.docs

No Local Indexing

A surprising architectural choice: Claude Code does not build an index of your codebase. No vector database, no embeddings, no background indexing process. Early versions experimented with RAG (Retrieval-Augmented Generation) using a local vector store, but the team found that agentic search — where the model uses Glob and Grep tools to actively explore the codebase — outperformed it significantly. This was confirmed by Boris Cherny, Claude Code's creator.src

The practical implication: every search starts from scratch against the live filesystem. For the vast majority of codebases (under 100K files), this works well and has the advantage of never serving stale results from an outdated index. For truly massive monorepos, it can be slower.

Prompt Caching

Here is an efficiency trick that matters: Anthropic reports a 92% prompt prefix reuse rate for Claude Code sessions, resulting in approximately 81% cost reduction.src Because the beginning of each API call (system prompt, CLAUDE.md, tool definitions, early conversation) is identical to the previous call, the API can cache and reuse the processing of that prefix. You pay full price only for the new content at the end of each turn.

07 — Glossary

The Technical Terms, Decoded

Every field has its jargon. Here is what the key terms actually mean, stripped of mystique.

Inference

+

The process of running an input through a trained AI model to produce an output. When Claude reads your code and suggests a fix, the "thinking" step is inference. It requires specialised hardware (GPUs/TPUs) and runs on Anthropic's servers, not your machine. Think of it as the model "using" its training — learning happened months ago, inference is applying that knowledge now.

Context Window

+

The total amount of text (measured in tokens) that the model can consider in a single API call. Claude Code uses up to 200K tokens per turn. Everything the model needs to know — your question, the conversation history, file contents, tool definitions — must fit within this window. It is not persistent memory; the model re-reads the full context on every turn.

Tokens

+

The units AI models use to measure text. A token is roughly 3/4 of an English word, or about 4 characters. "Hello world" is 2 tokens. A 500-line source file might be 2,000–5,000 tokens depending on density. Context windows, pricing, and rate limits are all measured in tokens.

Agentic Loop

+

A pattern where the AI model does not just answer a question — it takes actions (reading files, running commands), observes the results, and decides what to do next, in a continuous loop. This is what makes Claude Code different from a chatbot: it can independently investigate a bug, try a fix, run tests, and iterate until the problem is solved.

Tool Use

+

The mechanism by which the AI model requests actions to be performed. Instead of printing "you should run npm test," the model emits a structured tool_use block that says "run this command." The local client then executes it and returns the result. The model decides what to do; the local client does it. This separation is fundamental to Claude Code's architecture.

Bun

+

A JavaScript/TypeScript runtime, similar to Node.js but designed for speed. Claude Code chose Bun over Node.js for faster startup times and built-in TypeScript support. When you see a Claude Code process in your Activity Monitor, it is a Bun process, not a Node.js one. Bun is open-source, created by Jarred Sumner, and written in Zig.

Server-Sent Events (SSE)

+

A web standard for streaming data from a server to a client over HTTP. When Claude's response appears word-by-word in your terminal, that is SSE at work — the server sends chunks of the response as they are generated, rather than waiting for the full response to be complete. This is what makes Claude Code feel responsive even when generating long answers.

Compaction

+

The process of condensing the conversation context when it approaches the context window limit. Claude Code automatically summarises older parts of the conversation and discards detailed tool outputs that have already been processed. Think of it as the difference between a ball-by-ball commentary and a session summary — the key information is preserved, but the granular detail is compressed.

Zero Data Retention (ZDR)

+

A data handling mode where Anthropic does not store your API inputs or outputs at rest after the response is returned. Available for Enterprise and API (commercial) plans. Even under ZDR, content flagged for usage policy violations may be retained for up to 2 years. ZDR is not available on consumer plans (Free, Pro, Max).docs

SOC 2 Type II

+

An auditing standard that evaluates how a company manages customer data, specifically examining security, availability, processing integrity, confidentiality, and privacy controls. "Type II" means an independent auditor has verified that these controls were not only designed well but also operated effectively over a period of time (typically 6–12 months). It is the most widely recognised trust certification in cloud computing.

MCP (Model Context Protocol)

+

An open standard (created by Anthropic) that lets AI models connect to external tools and data sources through a standardised interface. In Claude Code, MCP servers can provide additional tools beyond the built-in ~40 — for example, connecting to databases, cloud services, or custom APIs. The VS Code extension uses MCP internally to bridge between VS Code and the CLI engine.

ripgrep (rg)

+

A blazing-fast text search tool written in Rust. When Claude Code searches your codebase (the Grep tool), it uses ripgrep under the hood, not the standard Unix grep. ripgrep is significantly faster, respects .gitignore files by default, and handles Unicode correctly. Claude Code bundles its own copy via the @vscode/ripgrep package.

Prompt Caching

+

An optimisation where the API reuses the processing of repeated prompt prefixes. Since the beginning of each Claude Code API call (system prompt, CLAUDE.md, tool definitions, early conversation) is identical to the previous call, the API can skip re-processing that prefix. Anthropic reports 92% prefix reuse in Claude Code sessions, resulting in ~81% cost reduction.
